# SceneHub Node v2 Transition Plan

This plan defines how the current SceneHub Node v1 firmware moves toward Node
v2 without turning the node into a second SceneHub controller.

The main product goal is:

- keep the node controllable from SceneHub;
- allow the node to run local standalone logic from an uploaded JSON bundle;
- expose the resulting capabilities and events back to SceneHub through
  `device_description`;
- keep memory, locking and hardware safety predictable.

## Target Modes

The firmware should support three runtime modes.

### `scenehub`

Normal managed mode.

- Wi-Fi and MQTT are required.
- SceneHub sends commands through the existing `cp/v1` device-control contract.
- Local v2 rules may be disabled, or limited to safety/interlock behavior.
- `describe_interface` reports base hardware capabilities and rule support.

### `standalone`

Local autonomous mode.

- SceneHub is not required for local logic.
- Wi-Fi may still be used for local web UI and diagnostics.
- An active v2 standalone bundle owns local quest logic.
- The node may still expose status and events over MQTT when a hub is present,
  but the local rule engine remains authoritative for standalone behavior.

### `fallback`

Hybrid mode.

- SceneHub normally controls the node.
- If the hub/MQTT connection is unavailable for a configured timeout, the node
  activates a local fallback bundle.
- When the hub returns, policy decides whether to stay in fallback until manual
  reset or return to SceneHub-managed operation.

The current alpha firmware already includes the first owner-driven `fallback`
runtime slice, but it is not yet field-stable or production-ready.

## Two JSON Contracts

Node v2 uses two separate JSON concepts.

### Public capability JSON: `device_description`

This is generated by firmware and returned to SceneHub from
`describe_interface`.

It describes what the node can do now:

- physical resources;
- base hardware commands;
- base hardware events;
- bundle-exported scenario commands/events;
- active rule bundle metadata;
- supported rule schema versions;
- current operation mode.

Operators should not hand-edit this JSON. It is an exported contract.

### Private behavior JSON: `standalone_bundle`

This is uploaded by an admin/operator or generated with AI assistance.

It describes how the node behaves locally:

- triggers;
- conditions;
- local state;
- phases;
- actions;
- emitted events;
- safety limits.

The node validates and compiles this JSON before activation. Runtime execution
must not traverse the raw JSON tree.

Detailed engine semantics are frozen in `NODE_V2_ENGINE_CONTRACT.md`.

Driver rollout is tracked in `NODE_V2_DRIVER_PLAN.md`.

## Current Implementation Snapshot

Current alpha baseline on top of the original plan:

- `scenehub`, `standalone` and an alpha first-slice `fallback` runtime are all
  implemented. `fallback` still requires broader hardware/fault confidence
  before it should be treated as field-stable.
- Standalone bundle export metadata is projected back into
  `describe_interface`, so SceneHub can prefer exported scenario-facing
  command/event names over raw channel names.
- Provisioning/admin/routes were split into narrower owner files instead of one
  large mixed config/admin/rules/NFC/LED file.
- Runtime/status/manifest/admin readers consume copied runtime DTOs through
  `runtime_snapshot_capture(...)` rather than borrowed internal snapshot
  pointers.
- Bundle store/apply/clear flows explicitly report reboot-required semantics,
  because activation still happens on restart.
- Local rule execution is prevented from re-entering exported MQTT rule
  command dispatch through `node_control`.

Remaining near-term work is product hardening, NFC product slice completion,
GM/node admin UX verification and fallback fault/soak testing.

## Bundle Export Projection

Node v2 should expose not only raw hardware capabilities but also a
bundle-defined scenario layer.

Example operator intent:

- physical output: `relay_2`
- scenario-facing command in SceneHub: `open_room_2`

The important rule is:

- firmware does not rename the physical relay internally;
- the active bundle may export a higher-level command/event surface for
  SceneHub;
- `device_description` should project that exported surface as the preferred
  scenario-facing contract.

This lets SceneHub work with room meaning instead of raw board channels.

Expected split:

- base hardware capabilities remain discoverable for diagnostics/admin and for
  nodes without a bundle;
- bundle-exported commands/events become the main scenario-facing controls for
  the active bundle;
- SceneHub may hide or de-emphasize claimed raw outputs in the normal device
  control list when a bundle exports a more meaningful control on top of them.

Initial bundle-export direction:

- `exports.commands[]` declares scenario-facing commands such as
  `open_room_2`;
- `exports.events[]` declares scenario-facing event metadata such as
  `room_2_opened`;
- exported commands route into the local rule engine through validated runtime
  command entry, not by bypassing `node_control`;
- exported events remain explicit `emit_event` outputs, not implicit leakage of
  every internal phase/timer/local event.

This preserves a clean architecture:

- hardware layer stays firmware-owned;
- scenario meaning stays bundle-owned;
- SceneHub reads one exported contract from `describe_interface`.

## Engine Direction

Node v2 should implement a compiled reactive state machine, not a general
script runtime.

The engine model is:

- event-driven;
- phase-aware;
- timer-based for delayed continuation;
- stateful through bounded local variables;
- branch-capable through declarative conditions and bounded action selection.

This means complex local logic such as `if`, `or`, reset paths, timed waits and
multi-step flows is allowed, but only through:

- `all` / `any` / `not`;
- `phase_is` / `set_phase`;
- `start_timer` / `cancel_timer` / `timer`;
- bounded `sequence`;
- bounded `choose`.

The engine should not support:

- loops;
- recursion;
- arbitrary expression strings;
- blocking waits;
- script VMs such as Lua or JavaScript.

If authoring input describes "wait until the next input happens", compile and
runtime should model that as a phase change plus later event or timer, not as a
blocked stack frame.

## Module Plan

Add Node v2 as several small modules instead of one large feature module.

### `node_runtime_mode`

Owns the selected operation mode and mode transitions.

Responsibilities:

- load mode from config;
- expose mode in status and `device_description`;
- decide whether MQTT is required;
- decide whether v2 rules run;
- apply safe output policy during mode changes.

### `node_rule_store`

Owns durable rule bundle storage.

Responsibilities:

- store raw uploaded bundle;
- store last-known-good metadata;
- clear bundles on factory reset;
- keep previous active bundle if a new write or activation fails.

### `node_rule_schema`

Owns bounded validation of the uploaded JSON.

Responsibilities:

- validate schema version;
- validate required fields;
- reject unknown trigger/condition/action kinds;
- validate resource logical names;
- validate exported command/event ids, labels and claimed resources;
- validate rule limits;
- validate command args before compile.

### `node_rule_compile`

Converts validated JSON into fixed runtime tables.

Responsibilities:

- resolve logical names such as `relay_1` to command args such as
  `channel: 1`;
- build trigger, condition, action and timer tables;
- compile bundle-export metadata for later `device_description` projection;
- lower author-facing branch/wait constructs into phase/timer driven runtime
  entries;
- build bounded string/value pools;
- reject bundles that do not fit fixed capacity.

### `node_rule_engine`

Runs compiled rule tables.

Responsibilities:

- consume input/timer/boot/MQTT events;
- evaluate triggers and conditions;
- execute actions under a per-tick budget;
- keep delayed continuation non-blocking by using timers and later events;
- publish diagnostics on limit failures;
- never allocate or parse JSON in the hot path.

### `node_action_router`

Maps rule actions to existing local commands.

Responsibilities:

- build a bounded args DTO or JSON scratch for `node_control_submit()`;
- preserve existing command validation;
- prevent rules from touching hardware drivers directly.

### `node_event_router`

Maps input, timer, boot and optional MQTT events into rule-engine events.

Responsibilities:

- poll or receive input changes;
- debounce/hold detection where owned by v2;
- publish rule-emitted events through the normal node event path when MQTT is
  available.

### `node_rule_api`

Admin/config API for rules.

Initial commands/routes:

- `node.rules.validate`
- `node.rules.apply`
- `node.rules.get`
- `node.rules.clear`
- `node.rules.pause`
- `node.rules.resume`

HTTP provisioning routes should call this API through the admin owner path, not
write storage directly.

### `node_ai_context`

Generates an AI-readable context document from the current firmware and pin
configuration.

Responsibilities:

- list logical inputs and outputs;
- list supported command actions;
- list allowed events;
- list hard limits;
- include a prompt template;
- never include secrets.

## Implementation Phases

### Phase 0: Documentation and Contract Freeze

- Update v2 design, schema draft and provisioning docs.
- Define operation modes.
- Define the two JSON contracts.
- Define initial firmware limits.
- Decide where rule bundles are stored.

### Phase 1: Runtime Mode Config

- Add operation mode to `node_config_t`.
- Add mode to local web UI.
- Allow `standalone` mode to initialize hardware/runtime without requiring a
  SceneHub controller host.
- Keep MQTT startup disabled or optional in standalone mode.

No rule execution yet.

Implementation status, 2026-06-16:

- `node_config_t` stores `operation_mode` with NVS config version `10`.
- Existing version `9` configs migrate to `scenehub` mode without changing pin
  or network settings.
- `node_runtime_mode` owns mode names and startup policy.
- Local provisioning API/UI can read and write `scenehub`, `standalone` and
  alpha `fallback`.
- `node.get_status` and `device_description.v2` expose the current mode.
- `standalone` no longer requires `controller_host` for provisioning
  completion and skips MQTT startup.
- Rule execution is intentionally still disabled.

### Phase 2: Rule Store and API Skeleton

- Add `node_rule_store`.
- Add validate/get/clear/apply API skeleton.
- Store raw bundle only after size and basic schema validation.
- Factory reset clears stored rule bundles.
- `apply` stores a validated candidate for next boot activation.

No rule execution yet.

Implementation status, 2026-06-18:

- `node_rule_store` persists raw bundle JSON and metadata in NVS under its own
  namespace.
- `node_rule_api` already exposes validate / get / apply / clear owner-facing
  entrypoints, plus runtime pause / resume controls.
- Local provisioning HTTP already exposes `GET /api/rules`,
  `POST /api/rules/validate`, `POST /api/rules/apply` and
  `POST /api/rules/clear`, plus `POST /api/rules/pause` and
  `POST /api/rules/resume`.
- Provisioning rule-edit routes now go through the admin owner task for
  validate / apply / clear / pause / resume instead of mutating runtime or
  storage state directly inside HTTP callbacks.
- Provisioning UI now has a `Rules` tab with stored-bundle metadata, JSON
  editor, format, validate, apply, pause, resume and clear actions.
- `apply` still stores the candidate for activation on reboot; runtime hot-swap
  is intentionally not part of this phase.
- Provisioning/admin heavy scratch owners now use a PSRAM-first allocation path
  with internal-RAM fallback instead of keeping the largest config/result
  buffers permanently in internal `.bss`.

### Phase 3: Schema Validation and AI Context

- Implement `node_rule_schema`.
- Add `/api/rules/context` or equivalent local UI export.
- Validate resources against current configured hardware.
- Validate commands against the capability registry.

Implementation status, 2026-06-17:

- `node_rule_schema` validates bundle JSON through bounded `cJSON` parsing.
- Validation checks top-level metadata, rule counts, action nesting, trigger /
  condition / action kinds and logical resource names against current
  `node_config_t`.
- Bundle `drivers[]` currently supports the first `nfc_reader` / `pn532` shape
  and validates known-card mappings into bounded local events.
- Provisioning `POST /api/rules/validate` already uses config-aware validation.
- Provisioning `GET /api/rules/context` now exports a first AI/authoring
  context snapshot for the current node: logical resources, supported engine
  constructs, command shapes, hard limits and prompt guidance.
- Provisioning context export now builds the `cJSON` tree outside the shared
  response-buffer lock and only takes the buffer lock for bounded print/send.
- The current AI-context export is intentionally descriptive only; it is not yet
  a full SceneHub admin API surface.

### Phase 4: Compile Without Execution

- Add `node_rule_compile`.
- Convert logical names to fixed command/action entries.
- Report compiled table usage in status.
- Keep activation disabled until compile is reliable.

Implementation status, 2026-06-17:

- Added initial `node_rule_compile` bootstrap stage.
- `validate` and `apply` now pass through compile, not only schema, so the node
  stores only a bundle that fits current fixed limits.
- On boot, stored rules are reloaded and compiled into fixed summary tables:
  rule list, trigger kinds, emitted events, state keys, timer names and phase
  names.
- `device_description.v2` now reports compiled rule status, active bundle
  metadata, standalone event names and compiled rule/action counters.
- Runtime execution is still not enabled; this is compile/status groundwork for
  the future engine.

### Phase 5: Minimal Rule Engine

First executable slice:

- `boot` trigger;
- `input_edge` trigger;
- `command` action for relay/mosfet/LED commands;
- `emit_event`;
- basic state set/check;
- `set_phase`.

All actions go through `node_control_submit()`.

Implementation status, 2026-06-17:

- Added `node_action_router` so rule-driven commands use the existing
  `node_control_submit()` path with source `local_rule`.
- Added `node_rule_engine` boot-time runtime owner with bounded local state and
  phase storage.
- Boot-triggered compiled rules can now execute `command`, `set_state`,
  `set_phase` and `emit_event` actions from fixed compiled tables.
- `node_rule_compile` now stores minimal runtime tables for supported triggers,
  conditions and actions instead of summary metadata only.
- Input polling ownership moved out of the old MQTT-only path into the rule
  engine owner task, so `input_edge` dispatch now also works in standalone mode
  without MQTT.
- The same input owner still publishes legacy `input.changed` events when MQTT
  is connected, preserving v1 observability.
- Added a public local-event dispatch entrypoint for future driver modules, so
  `local_event` rules already have a stable runtime injection path.
- Runtime-originated MQTT events now leave the rule path through a bounded
  action-router publish queue/owner task instead of publishing directly from
  the rule engine owner loop.
- The rule engine runtime config is now a compact owner DTO instead of a full
  copied `node_config_t`, keeping internal-RAM usage bounded to the fields the
  engine hot path actually needs.
- Current executable condition/action subset is intentionally smaller than the
  full schema draft: unsupported runtime constructs are rejected at compile
  time with explicit `runtime_*_not_supported` errors.

### Phase 6: Timers, Holds and Phases

- Add `input_hold`;
- add `start_timer` and `cancel_timer`;
- add `phase_is` and `set_phase`;
- add bounded action sequences;
- add bounded `choose` branching;
- keep waits modeled as timer/event continuation, not blocking delay.

Implementation status, 2026-06-17:

- `node_rule_compile` now lowers `timer` triggers plus `start_timer` /
  `cancel_timer` actions into fixed runtime tables with bundle-local timer
  names and bounded timer slots.
- `node_rule_engine` now owns fixed timer runtime state and dispatches
  `NODE_RULE_EVENT_TIMER` without parsing JSON or allocating in the hot path.
- Initial timer modes are wired as `oneshot`, `repeat` and `cooldown`.
- `input_hold` is now compiled and dispatched from the same input owner task as
  `input_edge`, so hold timing no longer depends on MQTT-only paths.
- Per-input debounce is now stored in base config for universal-input pins and
  used by the standalone input owner before `input_edge` / `input_hold`
  dispatch.
- Bounded `sequence` now compiles by flattening ordered nested actions into the
  fixed runtime action table.
- Initial bounded `choose` now executes in runtime for the currently supported
  compiled conditions, without adding a script VM or blocking waits.

### Phase 7: SceneHub Integration

- Extend `describe_interface` with v2 metadata.
- Project bundle-exported scenario commands/events into `device_description`.
- Publish `rules.changed` or status update after bundle activation.
- Let SceneHub upload/inspect/clear bundles as admin operations.
- Keep normal scenario commands separate from rule editing.

Planned bundle-export behavior:

- SceneHub `Devices -> Control` should prefer bundle-exported scenario controls
  such as `open_room_2`;
- raw physical controls such as `relay_2` remain available as base capability
  data, but may be hidden or de-emphasized when claimed by bundle exports;
- SceneHub `Devices -> Edit device` remains the home for rule admin operations
  such as validate/apply/clear/reboot.

Implementation status, 2026-06-18:

- `describe_interface` now exports v2 mode/runtime metadata, `rules.changed`
  event shape and a separate `admin_command_templates` section for
  `node.rules.*` operations plus `node.reboot`.
- Runtime status now exposes rule-engine state through `node.get_status`.
- SceneHub MQTT rule-admin operations now publish `rules.changed` and a fresh
  status update after successful apply / clear / pause / resume.
- MQTT transport now routes `node.rules.validate`, `node.rules.apply`,
  `node.rules.get`, `node.rules.clear`, `node.rules.pause` and
  `node.rules.resume`, plus `node.reboot`, through a dedicated admin path instead of
  `node_control_submit()`, keeping runtime scenario commands separate from
  rule-edit operations.
- MQTT status capability advertisement now includes the rule-admin commands and
  `rules.changed`.
- SceneHub-side UX is intentionally split:
  `command_templates` stay runtime/scenario-facing, while node admin operations
  stay outside the normal scenario/manual command list.
- Current intended Hub UX is:
  `Devices -> Edit device` owns the standalone-bundle admin panel and uses
  `node.rules.get` / `node.rules.validate` / `node.rules.apply`, plus
  `node.reboot`.
- `Device Controls` may expose only quick operational admin actions such as
  rules `pause`, rules `resume` and node `reboot`/`reset`; bundle-edit actions
  such as `validate` and `apply` must not be shown there as ordinary device
  control buttons.
- This split is part of the transport contract direction, not only a temporary
  UI preference: bundle editing is an admin workflow, not a scenario action or
  sidebar command.

Implementation update, 2026-06-20:

- SceneHub `Devices -> Edit device` now keeps the standalone rule-engine panel
  collapsed by default and shows inline operation status for `Get config`,
  import/save and node admin actions, so the modal itself exposes success or
  failure instead of relying only on the global page status line.
- Large admin payloads such as `node.rules.get` and wide
  `describe_interface`-derived metadata no longer require inflating ordinary
  steady-state ingest/result storage. Transient cached payload ownership is now
  separated from the small read-side summary path.
- Hub-side compact-manifest save failure now returns an explicit
  `device_manifest_too_large` error instead of failing as a silent generic
  persistence problem.
- Phase 7 is functionally in place. Remaining work after this point is product
  hardening, NFC product-slice completion and wider fallback field testing.
- The current mandatory Hub slice does not require centralized NFC card
  editing. For Node v2 baseline, SceneHub only needs:
  - reader health/status visibility;
  - exported NFC events/commands from `describe_interface`;
  - ordinary device/runtime status ingest.
- Node-local provisioning remains the current source of truth for NFC known
  cards and reader wiring/config. A future `node.nfc.cards.get/set` Hub admin
  path is product follow-up work, not a blocker for Phase 7 closure.

### Phase 8: Fallback Mode

- Add optional hub-offline timeout.
- Activate fallback bundle when SceneHub is unavailable.
- Define return-to-hub policy.

Implementation status, 2026-06-20:

- `node_fallback_runtime` now owns the first bounded fallback state machine
  and policy evaluation path instead of spreading hub-offline logic across MQTT
  and rule-engine modules.
- The runtime already honors configured offline timeout, return delay and
  return policy, and the current owner-model test app covers the baseline
  transition cases.
- This slice is still alpha only:
  - wider real-hardware fault testing is still required;
  - support should not treat fallback as production-ready;
  - follow-up work is hardening, not first implementation.

Current first-slice runtime contract:

- `fallback` has its own owner state machine, not ad-hoc `if` branches spread
  across MQTT and rule-engine modules.
- Initial states:
  - `hub_primary`
  - `hub_offline_pending`
  - `fallback_active`
  - `hub_return_pending`
- Initial inputs:
  - Wi-Fi connected / disconnected
  - MQTT connected / disconnected
- Initial derived condition:
  - `hub_available = wifi_ready && mqtt_connected`
- First-slice timing policy:
  - when `hub_available` becomes false, start `fallback_timeout_ms`;
  - when the timeout expires, enter local fallback runtime;
  - when `hub_available` becomes true again, start `fallback_return_delay_ms`;
  - when the return delay expires, leave local fallback runtime and return to
    hub-primary mode.
- First-slice return policy:
  - support only `auto_on_stable_mqtt`;
  - manual stay-in-fallback can remain a later extension.

Owner/runtime rules:

- fallback owner must use fixed queue items and static owner state;
- no JSON parsing or storage writes in the fallback owner loop;
- no MQTT transport lock may be held while entering or leaving fallback;
- entering fallback should use the rule-engine owner API, not mutate rule state
  tables directly;
- leaving fallback must execute a defined cleanup path instead of merely
  ignoring further local events.

First-slice baseline now in place:

1. `node_fallback_runtime` owns static queue/task/state and policy evaluation.
2. Wi-Fi and MQTT connectivity edges are routed into that owner.
3. Config fields exist for:
   - `fallback_timeout_ms`
   - `fallback_return_delay_ms`
   - `fallback_return_policy`
4. Fallback entry uses the rule-engine owner API.
5. Fallback exit uses the rule-engine owner API.
6. Provisioning and status surfaces already expose the current fallback tuning
   and runtime state.

Remaining Phase 8 work:

1. re-run real-hardware fault testing around Wi-Fi/MQTT loss and recovery;
2. confirm fallback never regresses unrelated node control/admin flows;
3. decide whether more return-policy variants are needed beyond the current
   first slice.

## Current Phase Summary

Status snapshot, 2026-06-20:

- Phases 1 through 7 are implemented as the current working Node v2 baseline:
  runtime mode separation, durable bundle storage, schema/compile/runtime,
  timers/holds/phases and the SceneHub admin/export split are all present.
- Phase 8 `fallback` is implemented as an alpha first slice, but it is not yet
  field-stable or production-ready.
- Near-term work is no longer "build the engine from scratch", but:
  - product-hardening the existing bundle/admin/export flow;
  - widening fallback hardware/fault verification before it graduates from
    alpha;
  - finishing the NFC product slice around local reader/cards/status/UI/export
    and health propagation;
  - keeping documentation/policies aligned with the implemented ownership
    model.
- Centralized NFC card management from SceneHub GM is explicitly outside the
  current baseline and should be tracked as a separate product follow-up, not
  treated as incomplete engine/runtime work.

## Safety Rules

- A bad bundle must never replace the active last-known-good bundle.
- Rule execution must never call hardware IO directly.
- Runtime rule execution must not allocate memory.
- Rule activation must be two-phase: validate/compile inactive candidate, then
  swap active tables.
- Complex waits must compile to timers/phases/events, not suspended rule
  frames.
- If an active rule exceeds budget, stop that rule, publish diagnostics and keep
  the node observable.
- Factory reset must clear user rule bundles and driver instance config.

## Open Decisions

- Storage backend for raw bundles: NVS blob, LittleFS or SD.
- Maximum raw bundle size for the first release.
- Maximum compiled rules/actions/timers.
- Maximum nesting depth for `sequence` and `choose`.
- Whether standalone mode still connects to MQTT when Wi-Fi and hub are
  configured.
- Whether fallback needs more return-policy variants beyond the current alpha
  baseline.
- Whether AI context is served only by local HTTP or also by SceneHub admin API.
